01 — Intent

Why this project exists

Most student portfolio projects are CRUD applications dressed up with a modern UI. Drishyamitra is designed to be different — a system where the interesting engineering is the point, not the wrapper around it.

"The most impressive thing in this project is not the face recognition. It is the agentic tool-calling layer — and most students building this will never get there."

The project targets three things that placement interviewers at AI-focused companies actually probe: an understanding of how LLMs behave as reasoning engines rather than text generators, the ability to design background processing systems, and the engineering discipline to build polyglot microservice architectures where each technology is justified by a concrete reason.

The core problem being solved

A person with 10,000 photos across devices cannot find a specific photo without scrolling. They cannot search by face. They cannot say "send my grandmother's birthday photos to my mother" and have it happen. Traditional photo apps solve storage. Drishyamitra solves retrieval and action — the two harder problems.

What a user experiences
Upload photos once. Faces are detected automatically. Label a person once and the system remembers forever. Ask in plain language: "Show me Dad's photos from Diwali 2023." The AI finds them. Say "Email them to mom." The AI does it.
What the system actually does
DeepFace extracts 512-dimensional face embeddings. Cosine similarity matches faces to known persons. A Groq-powered agent with tool-calling resolves multi-step natural language commands. BullMQ workers process jobs without blocking the API. Socket.io streams progress in real time.

Why these specific technologies

Every technology in this stack was chosen with a specific justification that can be defended in an interview. The rule is simple: if you cannot explain why you chose something over its alternatives, you should not have chosen it.

⚑
Design principle

This project demonstrates that you understand the why behind architectural decisions — not just the what. That distinction is the difference between a candidate who used tools and one who thought with them.

What this proves to an interviewer

  • You understand how LLMs work as reasoning engines with tool-calling capabilities, not just text generators
  • You can design background processing systems with proper queue management and failure handling
  • You can build polyglot architectures where the choice of language per service is technically justified
  • You understand the tradeoffs between synchronous and asynchronous processing
  • You can integrate computer vision models without fully owning the ML research behind them
  • You design for conversation state and reference resolution — not just single-turn queries
02 — Architecture

How the system is built

The architecture is a deliberate separation of concerns. Each layer does exactly one category of work. Nothing that could be done by a specialised tool is absorbed into a general-purpose layer.

React client (Vite)
Gallery UI, photo upload, face labeling with canvas overlay, AI chat interface, real-time progress via Socket.io. Axios for REST calls. React Context for auth state. Tailwind CSS for layout.
React Vite Tailwind CSS Axios Socket.io-client
Express API (Node.js)
REST endpoints for auth, photos, faces, chat, delivery, history. JWT middleware. Multer for file uploads. Calls Python face service via internal HTTP. Manages Redis session memory. Owns the agent loop.
Express.js JWT + bcryptjs Multer ioredis Socket.io
Groq agent layer
Llama-3.3-70b-versatile for complex multi-step tool chains. Llama-3.1-8b-instant for simple queries. A bounded agent loop (max 5 iterations) executes tool calls, appends results to message history, and calls Groq again until a final text response is produced.
Groq SDK llama-3.3-70b llama-3.1-8b Tool calling API
Redis (dual role)
BullMQ queue backend for face recognition and delivery jobs. Also stores conversation session memory per user — messages array, lastPhotoSearch, lastDelivery — with a 24-hour TTL. These two roles coexist cleanly in the same Redis instance with key namespacing.
Redis BullMQ Session memory 24hr TTL
Python face service (Flask)
A single-purpose microservice. POST /recognize accepts an image URL, runs RetinaFace for detection, FaceNet512 for 512-dim embeddings, and returns bounding boxes and embedding vectors. Node calls this service internally. Cosine similarity matching happens in Node after embeddings are returned.
Python + Flask DeepFace FaceNet512 RetinaFace OpenCV
MongoDB Atlas
Stores User, Photo, Person, Face (with 512-float embedding array), DeliveryHistory, and ChatHistory documents. Long-term persistent state. Face embeddings stored as float arrays — cosine similarity computed in Node for current scale. Upgrade path to Qdrant is clear and justified if scale demands it.
MongoDB Atlas Mongoose ODM 6 collections
Cloudinary + delivery services
Cloudinary stores original photos and serves optimised URLs. Nodemailer (Gmail SMTP) handles email delivery with photo attachments. whatsapp-web.js handles WhatsApp delivery via QR-authenticated Node session. All delivery actions log to DeliveryHistory.
Cloudinary Nodemailer whatsapp-web.js DeliveryHistory

The agent loop — the most important piece

Most students implement a chatbot: user sends message, LLM responds, done. An agent is different. It is a loop where the LLM can request tool execution, inspect the results, and decide whether to call another tool or produce a final response.

// The agent loop — runs inside Express route handler
async function runAgentLoop(userMessage, sessionId) {
  // Load session from Redis
  const session = await redis.get(`chat:${sessionId}`);
  session.messages.push({ role: 'user', content: userMessage });

  let response = await groq.chat.completions.create({
    model: routeModel(userMessage), // 8b or 70b
    messages: session.messages,
    tools: TOOL_DEFINITIONS,
    tool_choice: 'auto'
  });

  // Bounded loop — max 5 iterations to prevent runaway
  for (let i = 0; i < 5; i++) {
    if (response.choices[0].finish_reason !== 'tool_calls') break;

    const toolCalls = response.choices[0].message.tool_calls;
    session.messages.push(response.choices[0].message);

    for (const call of toolCalls) {
      const result = await executeTool(
        call.function.name,
        JSON.parse(call.function.arguments),
        session  // passes lastPhotoSearch for reference resolution
      );

      session.messages.push({
        role: 'tool',
        tool_call_id: call.id,
        content: JSON.stringify(result)
      });

      // Update structured state — powers "email them" resolution
      if (call.function.name === 'searchPhotos') {
        session.memory.lastPhotoSearch = {
          ...JSON.parse(call.function.arguments),
          photoIds: result.map(p => p._id)
        };
      }
    }

    response = await groq.chat.completions.create({
      model: 'llama-3.3-70b-versatile',
      messages: session.messages,
      tools: TOOL_DEFINITIONS,
      tool_choice: 'auto'
    });
  }

  session.messages.push(response.choices[0].message);
  // Trim to last 20 messages, save back to Redis with 24hr TTL
  session.messages = session.messages.slice(-20);
  await redis.setex(`chat:${sessionId}`, 86400, JSON.stringify(session));

  return response.choices[0].message.content;
}
◈
Why the structured session state matters

When the user says "email them to my brother," the model resolves "them" from conversation history. But the backend also has session.memory.lastPhotoSearch.photoIds. The email tool injects these IDs directly rather than trusting the LLM to reconstruct them. This is how you engineer around LLM unreliability — deterministic state augments probabilistic reasoning.

Database schema

// Face — stores embedding alongside identity reference

  photoId: ObjectId,    // → Photo
  personId: ObjectId,   // → Person (null if unlabeled)

// Session memory in Redis (not MongoDB)
{

      people: String[],
      dateRange: { start, end },
      photoIds: ObjectId[]          // injected into delivery tools

      recipient: String,
      medium: "email" | "whatsapp"
    }
  }
}
03 — Engineering decisions

Every choice, justified

The table below documents every major technology decision in the project, the alternatives considered, and the concrete reason for the choice. Know these before any interview.

Decision Choice Alternative Why v1?
Face recognition runtime Python (DeepFace) face-api.js FaceNet512 + RetinaFace accuracy is meaningfully better. Python is the right tool for this specific job. Polyglot framing is stronger on resume. Yes
Embedding storage MongoDB (float array) Qdrant / Pinecone At ≤10,000 faces, cosine similarity in Node is fast enough (≤20ms). Qdrant adds infrastructure complexity without solving a real problem at this scale. Yes
Vector DB Qdrant (v2 only) MongoDB embeddings Justified once scale demands it. Adding it to v1 is resume padding — interviewers will ask why and the answer "MongoDB was too slow" won't be true. v2
Session/chat memory Redis MongoDB Session memory is temporary (24hr TTL), frequently updated, and requires fast reads. MongoDB persistence overhead is unnecessary for ephemeral data. Yes
Background jobs BullMQ Celery, synchronous Node-native, Redis-backed, well-documented. Face recognition on batch uploads takes 2-10s per photo — synchronous processing blocks the API and ruins UX. Yes
LLM provider Groq OpenAI, Gemini LPU hardware delivers ~800 tokens/sec. Socket.io streaming feels genuinely real-time. Free tier sufficient for development and demo. OpenAI-compatible API. Yes
LLM model routing Dual model Single model Simple queries → llama-3.1-8b-instant (fast, cheap). Complex multi-step → llama-3.3-70b-versatile (reliable tool chaining). Model routing is a production pattern worth knowing. Yes
Real-time updates Socket.io Polling, SSE BullMQ workers need to push progress events to clients mid-job. Socket.io persistent connection enables server-push without client polling. Works with JWT auth. Yes
Photo storage Cloudinary AWS S3, local disk Free tier, CDN included, automatic image optimisation, simple Node SDK. S3 is industry standard but adds IAM complexity. Local disk doesn't survive deployment. Yes
Deployment Managed services Docker Compose + VPS Vercel + Railway + Atlas + Upstash handles 5 services with zero ops overhead. Docker orchestration across 5 services is a real time sink for a student project. Yes
UI animations Tailwind only Framer Motion Animations don't get you hired. Core engineering does. Framer Motion is a low-priority investment when the agent layer isn't solid yet. Skip

The one honest tradeoff

The Python microservice adds operational complexity — two languages, two runtimes, two deployment targets. This is a real cost. The justification is that DeepFace's accuracy advantage on difficult lighting and angle cases is concrete, measurable, and directly affects the product experience. In an interview, acknowledge the tradeoff explicitly. It shows engineering maturity to name the cost of your own decisions.

◎
The Qdrant question

If an interviewer asks "why not Qdrant?" — the answer is: "At my current scale of a few thousand faces, MongoDB cosine similarity is fast enough. Qdrant becomes justified when brute-force comparison latency exceeds acceptable thresholds, which for this project it does not. I've designed the embedding schema so migration to Qdrant requires no data changes — just a different query layer." This is a stronger answer than "I used Qdrant" without being able to explain HNSW indexing.

The tool set

Four tools are registered with the Groq agent. Each has a precise schema. Groq will only call tools whose parameter types match. Validation happens before execution.

05 — Interview preparation

Every question you will be asked

If you put this project on your resume, an interviewer at any AI-focused company will probe at least three of these questions. Know all ten before you submit applications.

Question 1
How does the agent resolve "them" in "email them to my brother"?

Conversation history is stored per session in Redis. Every Groq call receives the full message history, allowing the model to resolve references. Additionally, the backend maintains session.memory.lastPhotoSearch.photoIds — a structured list of the last search result IDs. The sendEmail tool injects these IDs directly rather than relying entirely on the LLM to reconstruct them from context. This hybrid approach is more reliable than pure LLM memory.
Question 2
Why Redis for session memory instead of MongoDB?

Session memory is temporary (24-hour TTL), frequently updated on every conversation turn, and requires sub-millisecond reads since it blocks the agent loop. MongoDB's persistence overhead and connection latency are unnecessary for ephemeral, session-scoped data. Redis is designed exactly for this pattern.
Question 3
Why BullMQ instead of processing synchronously?

Face recognition on a batch upload takes 2–10 seconds per photo depending on image resolution and number of faces. Processing synchronously blocks the Express event loop and forces the user to wait with no feedback. BullMQ moves the work to a background worker, the API returns immediately with a job ID, and Socket.io streams progress events as each photo is processed.
Question 4
Why a Python microservice instead of face-api.js?

DeepFace with FaceNet512 and RetinaFace produces measurably better embeddings than JavaScript alternatives, particularly in difficult lighting and off-angle conditions. Python is the correct tool for this specific problem. Node calls the service over HTTP internally — the polyglot boundary is clean and the Node team never touches Python code. This is a common production pattern, not a weakness.
Question 5
How do face embeddings work and how do you match faces?

FaceNet512 maps each face image to a 512-dimensional float vector where geometrically similar faces cluster close together. To identify a face, compute cosine similarity between its embedding and all stored embeddings for known persons. Cosine similarity is preferred over Euclidean distance because it measures angular difference regardless of vector magnitude — more robust across different image conditions.
Question 6
Why not Qdrant or another vector database?

At the current scale of a few thousand faces, cosine similarity in Node against MongoDB is fast enough — under 20ms for 5,000 embeddings. Qdrant's HNSW indexing becomes valuable at tens of thousands of vectors where brute-force latency exceeds acceptable thresholds. The embedding schema is designed for a clean migration — moving to Qdrant in v2 requires only a different query layer, no data changes.
Question 7
How do you handle tool-calling failures or LLM hallucinations?

All tool parameters are validated before execution — if Groq returns people: "dad" instead of people: ["dad"], it is coerced before the MongoDB query. If a tool throws, the error is returned to Groq as a tool result with an error message, allowing it to recover gracefully. The agent loop is bounded at 5 iterations to prevent runaway chains. Users receive a friendly message, not a raw error trace.
Question 8
How does Node communicate with the Python face service?

Via HTTP POST to the Python Flask service running on an internal port (localhost:8001 in development, Railway internal networking in production). The contract is simple: Node sends an imageUrl, Python returns a JSON array of face objects with embeddings and bounding boxes. The service has a health check endpoint that Node polls on startup and refuses to queue face jobs if it fails.
Question 9
How would you scale this if 10,000 users were using it simultaneously?

Horizontal BullMQ workers across multiple Node processes to handle recognition queue load. Redis cluster for session memory at scale. Move embeddings to Qdrant with HNSW indexing as face database grows past ~50,000. Cloudinary CDN handles photo serving at any scale without changes. Python face service scaled horizontally behind a load balancer. This is precisely when each deferred technology decision becomes justified.
Question 10
Walk me through what happens when I say "show Grandma's Diwali photos".

Message sent to POST /api/chat/message. Session loaded from Redis. Groq receives full message history + tool definitions. Returns finish_reason: tool_calls with searchPhotos({people:["Grandma"], event:"Diwali"}). Node executes: query Face collection for Grandma's personId, find Photos with those face IDs, optionally filter by event tag. Results returned to Groq as tool result. Groq generates final response. Session updated with photoIds in lastPhotoSearch. Response streamed to React via Socket.io.
